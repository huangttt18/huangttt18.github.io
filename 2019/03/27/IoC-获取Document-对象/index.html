该文章为学习笔记，收集整理于芋道源码，Spring源码深度解析

在loadDocument()方法的参数中有一个EntityResolver

它通过getEntityResolver()方法获取

  • 如果resourceLoader不为null，则创建一个ResourceEntityResolver

  • 如果resourceLoader为null，则创建DelegatingEntityResolver来将entityResolver委托给默认的 BeansDtdResolver 和 PluggableSchemaResolver

那么EntityResolver是做什么的？

EntityResolver

如果SAX应用程序需要实现自定义处理外部实体，则必须实现此接口并使用setEntityResovler方法向SAX驱动器注册一个实例。

即当解析一个XML，SAX首先应该读取XML文档的声明，根据声明寻找DTD定义，以便对文档进行一个验证。但是默认的寻找规则是通过网络来下载相应的DTD声明，并进行验证，这需要花费一段时间，而且如果过程中网络发生错误或不可用时，就会报错，因为DTD声明没有被找到。

EntityResolver的作用就是在项目本身提供一个寻找DTD声明的方法，比如将DTD声明文件放到项目中，在实现时直接读取文档并返回给SAX即可，这样就能避免网络下载发生错误的问题。

EntityResolver接口方法声明

  • publicId：被引用的外部实体的公共标识符，如果没有提供，则返回 null
  • systemId：被引用的外部实体的系统标识符

在这里EntityResolver的实现类有4个

  • ResourceEntityResolver
  • DelegatingEntityResolver
  • BeansDtdResolver
  • PluggableSchemaResolver

其中，DelegatingEntityResolver会调用BeansDtdResolver或者PluggableSchemaResolver来进行解析

  • 如果验证模式为DTD，那么调用BeanDtdResolver来进行解析
  • 如果验证模式为XSD，那么调用PluggableSchemaResolver来解析

BeanDtdResolver的解析过程

  • 检查systemId是否为空并且是否以.dtd结尾，如果不是，则使用默认的DTD声明的寻找模式
  • 解析DTD文件的文件名，并创建ClassPathResource，构造InputSource对象设置publicId、systemId并返回

PluggableSchemaResolver的解析过程

  • 检查systemId是否存在，如果存在，那么通过getSchemaMappings().get(systemId)获取resourceLocation
  • 检查resourceLocation是否为空，如果不为空，那么创建ClassPathResource，然后通过resource获取InputSource，设置publicId、systemId并返回
