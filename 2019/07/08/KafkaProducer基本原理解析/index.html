KafkaProducer基本原理剖析

KafkaProducer的整体架构

整个生产者客户端由两个线程协调运行，分别是主线程和Sender线程，其中主线程用于创建消息，Sender线程用于从ProducerAccumulator中获取消息并发送到Kafka中

KafkaProducer是线程安全的，可以在多线程中共享单个KafkaProducer实例，也可以将KafkaProducer实例池化供其他线程使用。

KafkaProducer创建消息

消息经过拦截器（非必须）、序列化器（必须）、分区器（非必须）的处理后将消息缓存到ProducerAccumulator中

消息拦截器

一般用于在消息发送前做一些准备工作，或者用于在发送回调逻辑前做一些定制化的需求。要实现自定义拦截器，我们需要实现ProducerInterceptor接口，接口中包含3个方法：

  • public ProducerRecord<String, String> onSend(ProducerRecord<String, String> producerRecord)，该方法用于对消息做一些定制化的需求
  • public void onAcknowledgement(RecordMetadata recordMetadata, Exception e)，该方法会在消息被应答之前或者消息发送失败时调用生产者拦截器的onAcknowledgement()方法，该方法运行与Producer的I/O线程中，所以逻辑最好简单明了，否则会影响消息发送的速度
  • public void close()，该方法用于做一些资源释放的动作

序列化器

生产者需要用序列化器将对象转换成字节数组才能通过网络发送给Kafka，对应的在消费者方，需要使用反序列化器将从Kafka中获取到的字节数组转换成相应的对象。

实现自定义序列化器，需要实现Serializer接口，此接口包括3个方法：

  • public void configure(Map<String, ?> configs, boolean isKey)，该方法用于配置当前类
  • public byte[] serializable(String topic, T data)，该方法用于执行序列化操作
  • public void close()，该方法用于释放资源，一般是空方法，如果实现了此方法，必须保证它的幂等性，因为Kafka可能会多次调用该方法

分区器

如果创建ProducerRecord时指定了partition字段，那么分区器就不是必须的了，如果没有指定，那么就需要依赖分区器来计算分区号

实现自定义分区器需要实现Partitioner接口，该接口包含了2个方法：

  • public int partition(String topic, Object key, byte[] keyBytes, Object value, byte[] valueBytes, Cluster cluster)，该方法用于计算分区号。
    • 分区号是依赖key来计算的，如果key != null，那么分区器会对key进行哈希（murmurHash2算法）来计算分区号，拥有相同key的消息会被写入同一个分区，如果key == null，那么消息会以轮询的方式发送到主题内的各个可用分区
    • 如果key != null，那么计算得到的分区号为所有分区中的任意一个
    • 如果key == null，那么计算得到的分区号仅为可用分区中的任意一个
  • public void close()

消息经过以上的一系列处理之后，会被缓存到RecordAccumulator中的双端队列（Deque）中，每个RecordAccumulator内部分别为每个分区维护了一个双端队列，队列的内容为ProducerBatch，即Deque<ProducerBatch>，消息写入时被追加到队列的尾部，Sender线程从队列的头部读取消息

Sender线程从ProducerAccumulator获取缓存的消息之后对消息进行进一步的处理之后将消息发送到Kafka

每个ProducerBatch中都包含了1个或者N个ProducerRecord。

每当一条新的ProducerRecord进入RecordAccumulator时，会先寻找与消息分区所对应的双端队列，接着从双端队列的尾部获取一个ProducerBatch，如果没有则新建，查看ProducerBatch是否还能写入这个ProducerRecord，如果可以则写入，如果不能则创建一个新的ProducerBatch，在新建ProducerBatch时会检查这条消息的大小是否超过batch.size参数，如果不超过，就可以创建batch.size大小的ProducerBatch，如果超过就创建消息大小的ProducerBatch，这段内存区域不会被复用。

Sender线程从RecordAccumulator中获取缓存的消息后，会将原本的<分区, Deque<ProducerBatch\>\>的保存形式转换为<Node, List<ProducerBatch>>形式，Node代表Kafka集群中的Broker节点

接着再转换为<Node, Request>形式，这样就可以将请求发送给各个Node了，Reuqest指的是Kafka的各种协议请求

请求在从Sender线程发往Kafka之前还会以Map<NodeId, Deque<Request>>的形式保存到InFlightRequests，InFlightRequests用于缓存了已经发送出去但是还没有收到响应的请求

Kafka消息的发送分为三种模式
  • 发后即忘（Fire-And-Forget），只管发送消息而不管消息是否到达
  • 同步（Sync）
  • 异步（ASync）
