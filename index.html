线程池

Executor框架

Executor的结构
  • 任务，被执行任务需要实现的接口，Runnable，Callable
  • 任务的执行，Executor和继承Executor接口的ExecutorService（Implements：ThreadPoolExecutor，ScheduledThreadPoolExecutor）接口
  • 异步计算的结果，Future接口和实现Future接口的FutureTask
Executor的组成成员

  • Executor，Executor框架的基础，将任务的提交与执行分离

  • ThreadPoolExecutor，用于执行被提交的任务

    corePoolSize：核心线程池大小

    maximumPool：最大线程池大小

    BlockingQueue：暂时保存任务的阻塞队列

    RejectedExecutionHandler：当ThreadPoolExecutor被关闭或者达到了最大线程池大小并且工作队列已满的情况下，execute()方法将要调用的Handler

    • FixedThreadPool
      • 创建固定数量的线程来执行任务，适用于负载较重的服务器
    • CachedThreadPool
      • 可以创建无限制数量的线程来执行短期异步任务，适用于负载较轻的服务器
    • SingleThreadPoolExecutor
      • 创建单个线程来顺序的执行各个任务，并确保不会有多线程并发的情况发生

  • ScheduledThreadPoolExecutor，用于周期性的执行被提交的任务

    • ScheduledThreadExecutor
      • 创建多个后台线程执行周期性任务
    • SingleThreadScheduledExecutor
      • 创建单个后台线程执行周期性任务，并保证各个任务的顺序执行

  • Future、FutureTask，异步计算的结果

  • Runnable、Callable，被执行任务需要实现的接口

线程的状态

通过查看Thread的源码，我们可以看到Thread拥有6种状态

为什么会有线程安全问题

  • 存在临界资源，也就是共享资源
  • 存在多个线程共同操作这些临界资源

为了解决线程安全问题，必须限制在同一时刻有且仅有一个线程能够操作这些临界资源，其他线程必须等到该线程处理完毕之后才能对临界资源进行处理

start()与run()的区别

start()

  • 调用JVM_StartThread方法创建线程并启动

run()

  • 普通方法，用于执行线程的业务逻辑

sleep()与wait()的区别

  • sleep是Thread类下的方法，wait是Object下的方法
  • sleep能在任何地方使用，wait方法只能在synchronized方法或者synchronized块中使用
  • sleep不释放锁，仅让出CPU，wait释放锁，同时让出CPU

synchronized

  • 如果多个线程同时访问一个对象的Synchronized method或者Synchronized block时，有且仅有一个线程能够同时访问这个method或block
  • 如果多个线程同时访问不同对象的Synchronized method或者Synchronized block时，互不干扰
  • 如果多个线程同时访问一个类的Synchronized static method或者Synchronized(类.class)时，有且仅有一个线程能够访问这个method或block
  • 如果多个线程同时访问不同类的Synchronized static method或者Synchronized(类.class)时，互不干扰
  • 如果两个或以上线程分别访问类的静态代码块和非静态代码块，两者互不干扰

早期Java中的synchronized是重量级锁，基于Mutex Lock实现，在获取锁的线程之间切换时要从用户态转换到内核态，开销较大

自旋锁

在大多数情况下，线程只需要等待很短的时间就可以获取到锁资源，因为这种短时间的等待而进行线程切换的消耗是不划算的，因此自旋锁出现了。

自旋锁通过让线程进入忙循环等待锁的释放，而不让度CPU，以此避免线程切换带来的高消耗

但是，如果锁被其他线程占用的时间较长，那么自旋锁的使用又会带来更多的消耗

所以自适应自旋锁出现了

自适应自旋锁
  • 自适应自旋锁的自旋次数不再固定
  • JVM通过分析前一次在该锁等待过程中自旋的时间已经锁的拥有者的状态来决定自旋的时间以及是否自旋
锁消除

Java代码通过JIT编译时，会对运行上下文进行扫描，去除不可能存在共享资源竞争的锁，节省无意义的锁请求时间

锁粗化

对于在短时间内频繁重复进行加锁、解锁操作的情况，JVM会对其进行锁粗化，即扩大加锁的范围

因为StringBuffer中的append方法是用synchronized修饰的，所以每次进行append操作都会对sb对象锁进行加锁、解锁操作，因此JVM会将对象锁粗化，将100次的加锁、解锁操作粗化为1次，即从while循环开始到循环结束

Synchronized的四种状态
  1. 无锁
  2. 偏向锁
    • 大多数情况下，锁不存在多线程竞争的问题，而总是由统一线程多次获得
    • 偏向锁的使用减少统一线程获取锁的代价
      • 如果一个线程获取了锁，那么锁就进入偏向模式，此时Mark Word的结构变为偏向锁结构，当该线程再次请求锁时，无需再次同步，只需要检查Mark Word的锁标记位置以及当前线程Id等于Mark Word的ThreadId即可
    • 不适用于锁竞争激烈的场景
  3. 轻量级锁
    • 加锁
      • JVM在当前线程的栈帧中创建用于存储锁记录的空间，并将对象头中的Mark Word复制到锁记录中（Displaced Mark Word）
      • 接着线程尝试使用CAS操作将对象头中的Mark Word替换为指向锁记录的指针，如果成功，则线程获得锁。如果失败，则表示有其他线程竞争锁，当前线程会尝试使用自旋锁来获取锁
    • 解锁
      • 通过CAS操作将Displaced Mark Word替换回对象头，如果成功，表示当前没有竞争发生，如果失败，则表示当前存在竞争情况，接着锁膨胀为重量级锁
  4. 重量级锁
  • 锁膨胀：无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁

进程和线程的区别

发表于 2019-03-22 | 分类于 多线程 ， Java
字数统计: 201 | 阅读时长 ≈ 1

区别

  1. 进程是一个独立的应用，线程不是独立应用
  2. 进程有独立的地址空间，相互不影响，线程只是进程的不同执行路径
  3. 线程没有独立的地址空间，多进程的程序比多线程程序健壮
  4. 进程的切换比线程切换开销大
  5. 进程是系统资源分配的基本单元，线程是CPU调度的基本单元
  6. 一个进程中有多个线程
在Java中
  • 每个运行中的程序至少有一个线程
  • 线程共享JVM中的堆，每个进程对应一个JVM实例
  • Java采用单线程编程模型，程序会自动创建主线程
  • 主线程可以创建子线程，原则上主线程后于子线程终止

MySQL中的锁机制

发表于 2019-03-22
字数统计: 621 | 阅读时长 ≈ 2

MySQL中的锁

  • 共享锁（读锁）：不阻塞，多个用户可以同时读一个资源，互不干扰
  • 排他锁（写锁）：一个写锁会阻塞其他的写锁和读锁，这样同一时间只允许一个用户进行写入操作，防止其他用户读取正在写入的资源

锁的粒度

  • 行锁：开销大，只锁定一行或几行数据
  • 表锁：开销小，锁定整张表

InnoDB支持行锁和表锁，MyISAM只支持表锁

悲观锁

指对数据被外界修改持保守态度，在处理的过程中，数据将处于锁定状态

乐观锁

悲观锁会对数据库性能造成较大影响，特别是对于长事务而言，因此出现了乐观锁。

乐观锁一般基于数据版本记录机制实现，即给数据添加一个版本号标记，读取数据时会将该版本号一起读出，如果数据发生了修改，那么就将版本号+1，再进行提交，提交时会将新的数据的版本与当前数据的版本号相比较，如果当前版本号小于新数据的版本号，那么予以更新，否则认为是过期数据

死锁

死锁发生在多个进程访问同一数据库时，而每个进程拥有的锁都是其他进程所需要的，因此造成所有的进程都无法继续进行下去。即，进程A等待进程B的锁，而进程B等待进程A的锁

发生死锁的必要条件
  1. 互斥条件，在一段时间内某个资源只允许一个进程占用
  2. 请求保持条件，进程已经至少拥有一个资源，又提出了新的资源需求，而该资源已经被其他进程占有，此时请求进程阻塞，但又对自己持有的资源保持不释放
  3. 不剥夺条件，进程已经获得的资源在未使用完毕前，不能被剥夺，只能在用完之后自己释放
  4. 环路等待条件，发生死锁时，必然有多个进程相互等待对方释放资源的情况发生
如何避免死锁
  1. 设置获得锁的超时时间
  2. 按顺序访问同对象
  3. 避免事务中的用户交互
  4. 保持事务简短并在一个批处理中
  5. 使用低隔离级别
  6. 使用绑定连接

MySQL索引

发表于 2019-03-22 | 分类于 数据库
字数统计: 2.3k | 阅读时长 ≈ 8

索引的好处

  1. 提高数据检索的速度，降低数据库IO成本，通过缩小表中需要查询的记录的数目从而加快搜索的速度
  2. 降低数据排序的成本，降低CPU消耗，索引查询快的一个原因是因为索引会先将数据排好序

索引的缺点

  1. 索引会占用存储空间，因为索引的本质是一张记录了主键与索引字段的表，通常以文件形式存储在硬盘中
  2. 索引的使用会降低表的更新速度，一旦表的数据发生了变化，对应的索引也需要一起变更

索引类型

  1. 普通索引，没有任何约束的索引
  2. 主键索引，特殊的唯一索引，不允许空值
  3. 唯一索引，具有唯一性约束的索引
  4. 复合索引，多个列组合在一起创建索引
  5. 外键索引，InnoDB类型的表才能创建外键索引，用于保证数据一致性，完整性和实现级联操作
  6. 全文索引，MySQL自带的全文索引只能用于 InnoDB、MyISAM ，并且只能对英文进行检索

索引创建的原则

  1. 最适合索引的列是出现在 WHERE 子句中的列，或连接子句中的列，而不是出现在 SELECT 关键字后的列。
  2. 索引列的基数越大，索引效果越好。（索引基数：索引中不含重复的索引值的数量，比如，1、2、3、3、4、7、7，它的索引基数为5，索引基数越大，表示重复数据越少，那么工作效果就越好）
  3. 根据情况创建复合索引，复合索引可以提高查询效率。（复合索引中，重复数据更好，索引基数会更大）
  4. 避免创建过多的索引，索引会额外占用磁盘空间，降低写操作效率。
  5. 主键尽量选择比较短的数据类型，可以有效减少索引的磁盘占用提高查询效率
  6. 对字符串进行索引，应该定制一个前缀长度，可以节省大量的索引空间。

索引使用注意事项

  1. 在where子句中避免使用!=、<>操作符，因为这会使得引擎放弃使用索引而进行全表扫描
  2. 在where子句中避免使用or来连接条件，理由同上
  3. 在where子句中避免对字段进行表达式操作，理由同上
  4. 在where子句中避免对字段进行函数操作，理由同上
  5. 不要在 WHERE 子句中的 = 左边进行函数、算术运算或其他表达式运算，否则系统将可能无法正确使用索引。
  6. 复合索引遵循前缀原则。
  7. 如果 MySQL 评估使用索引比全表扫描更慢，会放弃使用索引。如果此时想要索引，可以在语句中添加强制索引。
  8. 列类型是字符串类型，查询时一定要给值加引号，否则索引失效。
  9. LIKE 查询，% 不能在前，因为无法使用索引。如果需要模糊匹配，可以使用全文索引。

  2. B-Tree索引

什么是B-Tree索引

B-Tree是为磁盘等外部存储设备设计的一种平衡查找树。

磁盘
  • 系统从磁盘读取数据到内存时是以磁盘块（block）为基本单位的，位于同一个磁盘块中的数据会被一次性读取出来
  • InnoDB中有Page的概念，page是磁盘管理的最小单位，InnoDB 存储引擎中默认每个页的大小为 16 KB，可通过参数 innodb_page_size 将页的大小设置为 4K、8K、16K
  • 系统的磁盘块通常没有这么大，因此每次InnoDB申请磁盘空间时都会是多个连续的磁盘块，以达到每个page的大小，在查询时如果一个页中的数据都能有助于定位数据记录的位置，这将会减少磁盘 I/O 次数，提高查询效率。
B-Tree
  • 特性
    • 每个节点最多有m个孩子
    • 除了根节点和叶子节点外，其他每个节点至少有ceil(m/2)个孩子
    • 若根节点不是叶子节点，则至少有2个孩子
    • 所有的叶子节点都在同一层，且不包含其他关键字信息
    • 每个非叶子节点包含 n 个关键字信息（P0,P1,…Pn, k1,…kn）
      • 关键字的个数 n 满足：ceil(m/2)-1 <= n <= m-1
      • ki(i=1,…n) 为关键字，且关键字升序排序。
      • Pi(i=0,…n) 为指向子树根节点的指针。P(i-1) 指向的子树的所有节点关键字均小于 ki ，但都大于 k(i-1)

  • B-Tree每个节点都是一个磁盘块，一个节点有2个升序排序的key和3个指向子树根节点的point，两个key将节点划分为3个区域，分别对应3个point指向的子树的数据范围，例如p1 k1=20 p2 k2 = 40 p3，那么p1所指向的子树的数据范围就是0-20，p2[20-40]，p3[40-…]
B+Tree

B+Tree是在B-Tree的基础上进行优化后的存储索引结构，更适用于外部存储

  • 在B-Tree中，每个节点包括的数据有，key，point和对应的data
  • 因为每页的存储空间是有限的，如果data较大，那么每个节点能存储的key的范围就很小，当存储数据量很大时同样会导致B-Tree的深度增加，因而增加磁盘的I/O次数，进而影响查询效率
  • 而在B+Tree中，所有数据记录节点都是按照键值大小顺序存放在同一层的叶子节点上，而非叶子节点上只存储 key 值信息，这样可以大大加大每个节点存储的 key 值数量，降低 B+Tree 的高度。
B+Tree和B-Tree的区别
  • 在B+Tree中，非叶子节点只存储键值，叶子节点才存储data
  • 所有的叶子节点中都有一个链指针
  • 数据记录都存放在叶子节点中
B+Tree中的索引类型
  • 主键索引（聚簇索引），存储真正的数据
    • 聚簇索引最大限度的提高了I/O密集型应用的性能，但是它也有限制
      • 插入速度严重依赖于插入顺序，按照主键的顺序插入是最快的方式，否则将会出现页分裂，严重影响性能，因此在InnoDB表中一般都会定义一个自增的ID列为主键
      • 更新主键会导致被更新的行移动，因此对于InnoDB表，我们一般定义主键为不可更新（MySQL默认是可以更新的）
      • 二级索引访问需要两次索引查找，第一次找到主键值，第二次根据主键值找到行数据。
      • 主键 ID 建议使用整型。因为，每个主键索引的 B+Tree 节点的键值可以存储更多主键 ID ，每个非主键索引的 B+Tree 节点的数据可以存储更多主键 ID 。
  • 非主键索引（辅助索引），存储整行数据的主键，键值是索引

通过辅助索引来查询数据时，需要两步：

  1. 通过辅助索引查询到对应的主键
  2. 再通过主键在聚簇索引中找到完整的数据行
索引的最左匹配原则

当索引为（name,age,sex）时，B+Tree是从左到右的来建立搜索树的

  • 比如当 (张三, 20, F) 这样的数据来检索的时候，B+Tree会优先比较name来确定下一步的搜索方向，然后比较age，再比较sex，最后得到检索的数据
  • 如果数据是这样的 (20, F) ，那么B+Tree就不知道该如何查找节点，因为B+Tree必须从最左边的索引开始才能建立搜索树
  • 当 (张三, F) 这样的数据来检索时，B+Tree 可以用 name 来指定搜索方向，但下一个字段 age 的缺失，所以只能把名字等于张三的数据都找到，然后再匹配性别是 F 的数据了。
MyISAM索引实现

MyISAM也是使用B+Tree，但是MyISAM的索引文件和数据文件是分离的，索引文件中仅保存了数据记录的地址。

主键索引

MyISAM叶节点的data域存放的是数据记录的地址，key不能重复

辅助索引

辅助索引的 key 可以重复

MyISAM 的索引方式叫做“非聚集”的

MyISAM 索引与 InnoDB 索引的区别？
  • InnoDB是聚簇索引，MyISAM是非聚簇索引
  • InnoDB主键索引中的叶子节点存储的是数据，索引非常高效，MyISAM存储的是数据的地址
  • InnoDB 非主键索引的叶子节点存储的是主键和其他带索引的列数据，因此查询时做到覆盖索引会非常高效。

什么是事务

  • 事务就是对一系列的数据库操作进行统一提交或回滚操作，要么全部成功，要么全部失败，不允许部分成功部分失败的情况

事务的特性

ACID

  • A：原子性，一个事务（transaction）中的所有操作，或者全部完成，或者全部不完成，不会结束在中间某个环节。事务在执行过程中发生错误，会被恢复（Rollback）到事务开始前的状态

  • C：一致性，在事务开始之前和事务结束以后，数据库的完整性没有被破坏。

  • I：隔离性，数据库允许多个并发事务同时对其数据进行读写和修改的能力，隔离性可以防止多个事务并发执行时由于交叉执行而导致数据的不一致。事务分为四种隔离级别

  • D：持久性，事务处理结束后，对数据的修改就是永久的，即便系统故障也不会丢失。

事务的并发操作

  • 事务的并发操作会导致三个问题：

    1. 脏读，事务A读取了事务B修改的数据，然后事务B进行回滚操作，那么A读取到的数据就是脏数据
    2. 不可重复读，事务A多次读取同一数据，在这个过程中事务B修改了该数据，并进行提交，导致事务A多次读取的同一数据结果不一致
    3. 幻读，事务A修改表中所有数据的某一个字段的值，在这个过程中，事务B新增了一条没有被修改过字段的数据，当事务A完成之后发现还有数据没有被修改，即幻读

事务的四种隔离级别

  • READ UNCOMMITED：事务中的修改即使没有被提交，对其他事务也是可见的，会导致脏读、不可重复读和幻读
  • READ COMMITED：事务从开始直到提交之前，所做的任何修改对其他事务都是不可见的，会导致不可重复读和幻读
  • REPEATABLE READ：一个事务按相同的查询条件读取以前检索过的数据，其他事务插入了满足其查询条件的新数据。产生幻行。导致幻读（MySQL默认的事务隔离级别）
  • SERIALIZABLE：强制事务串行执行

Linux常用命令总结

查找特定文件

  • find path [options] param - find ~/Downloads -name “redis-5*”
  • 在指定目录查找文件

对文件内容做统计

  • awk [options] ‘cmd’ file
  • 一次读取一行文本，按输入分隔符进行切片，切成多个部分
  • 将切片直接保存在内建的变量中，\$1, \$2,\$0表示行的全部内容
  • 支持对单个切片的判断，支持循环判断，默认分隔符为空格

检索文件内容

  • grep [options] pattern file
  • 查找文件中符合条件的字符串
  • grep ‘partial\[true\]’ filename，在指定的文件中查找包含’’中字符串的内容
  • grep -o ‘engine\[[0-9a-z]\]’，筛选出符合该正则表达式的内容
  • grep -v ‘grep’ 筛选掉包含’’中字符串的内容

批量替换文本内容

  • sed [options] ‘sed command’ filename
  • 适用于对文本的行内容进行处理
